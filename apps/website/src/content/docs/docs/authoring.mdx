A blueprint is just a TypeScript module that exports functions returning `Blueprint` values.
Anyone who can write Effect can write a blueprint.

## Your first blueprint

A region-mode blueprint that adds a section to `README.md`. Reach for `markdownSection` â€”
the SDK helper that bakes in the HTML comment delimiters so the markers don't render in the
published file:

```ts
import { markdownSection } from "@projitect/blueprint"
import type { Blueprint } from "@projitect/core"

export const myReadmeSection = (): Blueprint.Blueprint =>
  markdownSection({
    id: "my-org:readme:intro",
    version: "1.0.0",
    description: "Intro section for org-standard README",
    path: "README.md",
    content: "This project is part of My Org. See https://my-org/handbook.\n",
  })
```

Project maintainers consume it via:

```ts
import { pjt } from "projitect/cli"
import { myReadmeSection } from "@my-org/readme-blueprints"

export default pjt({ blueprints: [myReadmeSection()] })
```

That's the whole API for static content blueprints.

## Section helpers

Two thin wrappers over `regionFile` are exported for the two most common region shapes â€”
they exist because the names document intent and the right comment delimiters are baked in:

- **`markdownSection`** for HTML/MDX/XML files. Fences with `<!-- pjt:<id> start -->` and
  `<!-- pjt:<id> end -->`. The markers render invisible in markdown output, so READMEs stay
  clean for human readers. Use for `README.md`, `AGENTS.md`, `CONTRIBUTING.md`, MDX docs,
  any HTML/XML file.
- **`ignoreSection`** for `#`-comment ignore-style files. Fences with `# pjt:<id> start`
  and `# pjt:<id> end`. Use for `.gitignore`, `.eslintignore`, `.prettierignore`,
  `.dockerignore`, `.npmignore`.

```ts
import { ignoreSection, markdownSection } from "@projitect/blueprint"

ignoreSection({
  id: "my-org:eslintignore:generated",
  version: "1.0.0",
  path: ".eslintignore",
  content: "dist/\nbuild/\ncoverage/\n",
})

markdownSection({
  id: "my-org:agents:rules",
  version: "1.0.0",
  path: "AGENTS.md",
  content: "Always run `pnpm check-all` before merging.\n",
})
```

For other comment styles (`//`, `--`, `;`), drop down to `regionFile` directly and pass the
prefix you need.

## Computed blueprints (using Effect)

When you need to read state, branch on detected configuration, or compose multiple operations,
write the `plan` Effect yourself:

```ts
import { BlueprintFileSystem, type Blueprint, type ChangeSet } from "@projitect/core"
import { computed, readPackageJson } from "@projitect/blueprint"
import { Effect } from "effect"

export const reactConfig = (opts: { typescript: boolean }): Blueprint.Blueprint =>
  computed({
    id: "my-org:react:config",
    version: "1.0.0",
    permissions: [
      { kind: "read", glob: "package.json" },
      { kind: "write", glob: "src/index.tsx" },
      { kind: "write", glob: "src/index.jsx" },
    ],
    plan: Effect.gen(function* () {
      const fs = yield* BlueprintFileSystem
      const pkg = yield* readPackageJson()
      // ... compute the change set based on package state ...
      return { operations: [] } satisfies ChangeSet.ChangeSet
    }),
  })
```

You get the full Effect surface area: `Effect.if`, `Effect.when`, `Effect.forEach`, error
handling via `Effect.catchTag`, custom Layers for shared services, and so on.

## Permissions

Every blueprint declares what it needs:

```ts
permissions: [
  { kind: "read", glob: "package.json" },
  { kind: "write", glob: ".gitignore" },
  { kind: "write", glob: "src/**/*.ts" },
]
```

The `BlueprintFileSystem` rejects operations outside this set with `pjt.fs.permission-denied`.
**Declare what you need and nothing more** â€” project maintainers read this list when deciding
whether to install your blueprint.

Globs support `*` (any non-slash chars) and `**` (any chars including slashes). No brace
expansion in v0.

## directory() for scoped blueprints

To re-root a tree of blueprints under a subdirectory:

```ts
import { pjt, directory } from "projitect/cli"

export default pjt({
  blueprints: [
    directory("docs", [
      // Each child blueprint's paths are now relative to docs/, and their write
      // permissions are restricted to that subtree.
      myReadmeSection(),
    ]),
  ],
})
```

This is useful for monorepos where the same blueprint should apply to multiple packages.

## Testing

`@projitect/test-kit` exposes an in-memory `BlueprintFileSystem` Layer:

```ts
import { makeInMemoryLayer, dumpFs } from "@projitect/test-kit"
import { Effect } from "effect"
import { myReadmeSection } from "./blueprint"
import { it, expect } from "vitest"

it("adds a region to an existing README", async () => {
  const layer = makeInMemoryLayer({ "README.md": "# Project\n\nExisting content.\n" })
  const blueprint = myReadmeSection()
  await Effect.runPromise(blueprint.plan.pipe(Effect.provide(layer)))
  const fs = await Effect.runPromise(dumpFs(layer))
  expect(fs["README.md"]).toMatchSnapshot()
})
```

## Publishing

A blueprint package's `package.json`:

```json
{
  "name": "@my-org/readme-blueprints",
  "version": "1.0.0",
  "type": "module",
  "exports": { ".": "./dist/index.js" },
  "peerDependencies": {
    "@projitect/blueprint": "^0.0.0",
    "@projitect/core": "^0.0.0",
    "effect": "^4.0.0-beta.70"
  },
  "projitect": {
    "type": "blueprint",
    "import": "import { myReadmeSection } from \"@my-org/readme-blueprints\"",
    "call": "myReadmeSection()"
  }
}
```

Peer-dep `effect`, `@projitect/blueprint`, and `@projitect/core` â€” never bundle them. The
project maintainer's install resolves to a single shared copy.

### The `"projitect"` metadata field

Adding this field to your `package.json` lets `pjt add <your-package>` auto-splice your
blueprint into the user's `.pjt.ts`. Without it, `pjt add` still installs the package but
prints a "no metadata, edit `.pjt.ts` by hand" message.

Two shapes are supported:

**Single-blueprint** (one export, one call):

```json
"projitect": {
  "type": "blueprint",
  "import": "import { myBlueprint } from \"@my-org/my-blueprint\"",
  "call": "myBlueprint()"
}
```

**Blueprint-set** (multiple sections the user picks among, e.g. one per OS):

```json
"projitect": {
  "type": "blueprint-set",
  "import": "import { gitignores } from \"@projitect/blueprint-gitignore\"",
  "callTemplate": "gitignores.{section}()",
  "sections": ["macOs", "windows", "linux", "node", "next", "vscode", "jetbrains", "tsbuildinfo"]
}
```

`{section}` is substituted with each chosen section name. When the user runs
`pjt add @projitect/blueprint-gitignore --section macOs,node`, the splice inserts
`gitignores.macOs(),` and `gitignores.node(),` into the blueprints array.

If `--section` is omitted, **all** declared sections are added â€” useful for "starter kits"
where the user wants the whole bundle.

### The two marker conventions

`pjt add` splices into the user's `.pjt.ts` between two **convention markers**:

- `// pjt:imports start` / `// pjt:imports end` â€” for the import statement
- `// pjt:blueprints start` / `// pjt:blueprints end` â€” for the call lines

These are NOT lockfile-managed regions and NOT drift-checked. They're just splice anchors.
Don't confuse them with `// pjt:projitect:imports start/end`, which IS a managed region (owned
by the projitect blueprint) and IS drift-checked.

If a user has hand-edited `.pjt.ts` and removed the convention markers, `pjt add` fails with
[`pjt.add.markers-missing`](/errors/pjt.add.markers-missing/) and tells them how to restore them.

<Aside type="note" title="Lint rule">
  The `@projitect/blueprint` package ships an ESLint preset that forbids `node:fs` and
  `child_process` imports. Add it to your blueprint package's `eslint.config.js` so the soft sandbox
  boundary is enforced at lint time.
</Aside>
